## Claim 1 — Gemini's tokenizer

**As stated in `slides.md` (Gemini callout slide):**

> Gemini uses a SentencePiece tokenizer. The current public vocabulary is on the order of 256K pieces, shared with the open Gemma family. Google has published enough for Gemma users to reproduce the tokenization exactly, but not the full training procedure.

**Status:** confirmed (claims-ledger row 1).

**Sources:**
- [Understand and count tokens — Gemini API docs (Google AI for Developers)](https://ai.google.dev/gemini-api/docs/tokens)
- [Welcome Gemma 3 — Hugging Face blog (2025)](https://huggingface.co/blog/gemma3)
- [dejan.ai — Gemini tokenizer analysis (2024)](https://dejan.ai/blog/gemini-toknizer/)

**Pull quote** (Hugging Face / Gemma 3 release notes, describing the tokenizer shared with Gemini 2.0):

> "It is a SentencePiece tokenizer with 262K entries. The new tokenizer significantly improves the encoding of Chinese, Japanese and Korean text, at the expense of a slight increase of the token counts for English and Code."

**Why the hedge:** Google publishes the Gemma tokenizer (which is the same family) but does not publish the Gemini-specific training corpus or merge procedure. "On the order of 256K" is correct as a public-facing approximation; the Gemma 3 figure is 262K. The deck deliberately rounds rather than asserting an exact Gemini-specific number.

**What could change between now and delivery:**
- Google could publish a Gemini-specific tokenizer card (unlikely on the timeframe but watch the AI for Developers docs).
- Vocabulary size could shift in a future Gemma family release; verify "256K" is still the right ballpark on the week-of recheck.

## Claim 2 — Embedding / residual-stream dimensionality

**As stated in `slides.md` (residual-stream section):**

> GPT-3 used a 12,288-dimensional residual stream. Modern open models in the 7–13B class typically sit in the 4,096–5,120 range. Gemini's dimensions are not public.

**Status:** confirmed (claims-ledger row 2).

**Sources:**
- ["Language Models are Few-Shot Learners" — Brown et al., arXiv:2005.14165 (2020)](https://arxiv.org/abs/2005.14165) — the GPT-3 paper, Table 2.1.
- Secondary confirmation: [dugas.ch — "GPT-3 on a Napkin"](https://dugas.ch/artificial_curiosity/GPT_architecture.html)
- For "4,096–5,120": Llama 2 7B/13B model card (`hidden_size = 4096`); Mistral 7B model card (`hidden_size = 4096`); Llama 3 8B (`hidden_size = 4096`); Llama 3 70B (`hidden_size = 8192` — outside the 7–13B band, used as a sanity contrast only).

**Pull quote** (Brown et al., 2020, Table 2.1):

> "GPT-3 175B uses … d_model = 12288 and n_layers = 96"

**Why the hedge:** GPT-3 175B's 12,288 is exact. The "4,096–5,120" range is correct for the most-deployed open 7–13B-class models in 2025–2026, but other models in that size band exist with slightly different widths — the deck says "typically sit in" rather than "are." Gemini's residual-stream width is not in any public source the §10.1 sweep located.

**What could change between now and delivery:**
- Unlikely. The GPT-3 number is historical and fixed. The "modern open model" range is broad and stable.
- If a frontier-scale open-weight release lands the week before (e.g., a new Llama generation) and changes the typical range, update the slide; otherwise leave.

## Demo-data integrity check

The two interactive demos that ship with S1 use real model artifacts; verify these are still the artifacts the deck claims:

| Demo | Claim made in slides / demo footer | Verification |
|---|---|---|
| `demos/tokenizer-explorer/` | "Tokenizer: `cl100k_base` — the BPE used by GPT-4 family." | `gpt-tokenizer` npm package, version pinned in `package.json`. The package vendors `cl100k_base`'s merge tables. |
| `demos/embedding-scatter/` | "Real GPT-2 small (124M) input embeddings — one row of the `wte` table per word — projected to 2D with PCA." | Generated by `scripts/precompute_gpt2_data.py` from `transformers` `GPT2Model.from_pretrained('gpt2')`. Output `data.json` records `model: gpt2-small (124M)` and `source: wte (input-embedding) matrix`. PCA via centered SVD; explained variance reported in the deck. |
| `demos/residual-stream-animator/` | "Real activations from GPT-2 small (124M, 12 transformer blocks). Each block reads from the residual stream and adds something to it — never overwrites." | Same script, same pretrained `gpt2`, same precompute pipeline. Hidden states captured via `output_hidden_states=True`; per-layer norms and per-block delta norms persisted to `data.json`. The "GPT-2 applies a final layer norm before the LM head; we skip it" caveat is reproduced both in the demo footer and in the deck's residual-stream explainer. |
| `demos/agent-loop/` | "Hand-built scenario; the model traces and tool calls are illustrative, not from a live model run." | Scenario hand-authored in `src/scenario.js` to mirror the shape of a real SRE debugging trajectory. The deck does not claim these traces come from a live agent; it is explicitly framed as "what an agent loop looks like." |
